**Team card component (`app/(dashboard)/settings/components/team-card.tsx`):**

Create a "use client" component that provides full team management:

**Structure:**
- `Card` wrapper with `CardHeader` ("Team") and `CardContent`.
- Two sections: invite form at top, member list below.

**Invite form section:**
- Single row: email `Input` field, role `Select` dropdown (Admin/Member options using shadcn Select), "Invite" `Button`.
- On submit: call `sendInvite(email, role)` server action.
- Show success toast (use `sonner` if available, or inline success message) or inline error message.
- Clear form on success.

**Member list section:**
- On mount, call `getTeamMembers()` to load the list. Show `Loader2` spinner while loading.
- Display each member as a row in a clean list:
  - Email address (primary text)
  - Role badge: use inline `div+span` pattern from DESIGN.md (not Badge component). Admin = primary color, Member = muted color.
  - Status: "Active" for members, "Pending" with muted/amber badge for invites.
  - For active members: role change dropdown and remove button (both with confirmation dialogs).
  - For pending invites: "Resend" and "Cancel" action buttons.

**Role change:**
- Clicking the role badge or a dropdown triggers a `Dialog` (shadcn) with:
  - "Change role for {email}?"
  - Role select: Admin / Member
  - "Confirm" button calling `changeRole(userId, newRole)`.
  - On success: refresh the member list.

**Remove member:**
- "Remove" button (destructive variant or muted with red text) triggers a `Dialog`:
  - "Remove {email} from the team?"
  - "This will immediately revoke their access."
  - "Remove" button (destructive) calling `removeTeamMember(userId)`.
  - If error returned (last admin), show error in the dialog.
  - On success: refresh member list.

**Cancel invite:**
- "Cancel" button calls `cancelInvite(inviteId)`. No dialog needed (low risk action).
- On success: refresh member list.

**Resend invite:**
- "Resend" button calls `resendInvite(inviteId)`.
- On success: show brief "Invite resent" message, refresh list.

**State management:**
- `useState` for members array, loading state, invite form fields.
- `useCallback` for `refreshMembers` function that re-fetches the list.
- After any mutation (invite, remove, change role, cancel, resend), call `refreshMembers()`.

**Trial reminder email (`lib/billing/notifications.ts`):**

Add `sendTrialEndingSoonEmail()` function following the exact same pattern as `sendPaymentFailedEmail`:

```typescript
export async function sendTrialEndingSoonEmail(
  orgId: string,
  orgName: string,
  trialEndsAt: string,
  supabase: SupabaseClient
): Promise<void>
```

- Query admin user IDs from `user_organisations` WHERE `org_id = orgId AND role = "admin"`.
- Resolve emails via `supabase.auth.admin.getUserById()` (same pattern as payment-failed).
- Use platform `POSTMARK_SERVER_TOKEN`.
- Email content:
  - Subject: "Your trial for {orgName} ends in 3 days"
  - Body: "Your free trial for {orgName} ends on {formatted date}. To continue using the platform, please set up a subscription. [Upgrade Now]({billingUrl})"
  - Include both HTML and plain text versions following the same template structure as `sendPaymentFailedEmail`.
  - Use amber/warning color scheme in HTML header (not red — it's a reminder, not an error).
  - CTA button: "Upgrade Now" linking to `/billing`.
- Do not throw on individual email failures — log and continue.

**Trial reminder cron (`app/api/cron/trial-reminder/route.ts`):**

Follow the exact same pattern as `app/api/cron/trial-expiry/route.ts`:

- `export async function GET(request: Request)` handler.
- Verify `CRON_SECRET` from Authorization header (same auth pattern as other crons).
- Use admin client (`createAdminClient()`).
- Find orgs where trial ends in 3-4 days:
  ```typescript
  const now = new Date();
  const threeDays = new Date(now.getTime() + 3 * 24 * 60 * 60 * 1000);
  const fourDays = new Date(now.getTime() + 4 * 24 * 60 * 60 * 1000);

  const { data: orgs } = await adminClient
    .from("organisations")
    .select("id, name, trial_ends_at")
    .eq("subscription_status", "trialing")
    .gte("trial_ends_at", threeDays.toISOString())
    .lt("trial_ends_at", fourDays.toISOString());
  ```
- For each org, check idempotency via `app_settings`:
  ```typescript
  const { data: alreadySent } = await adminClient
    .from("app_settings")
    .select("id")
    .eq("org_id", org.id)
    .eq("key", "trial_reminder_sent")
    .eq("value", "true")
    .single();

  if (alreadySent) continue; // skip — already notified
  ```
- Call `sendTrialEndingSoonEmail(org.id, org.name, org.trial_ends_at, adminClient)`.
- After successful send, mark as sent:
  ```typescript
  await adminClient.from("app_settings").upsert(
    { org_id: org.id, key: "trial_reminder_sent", value: "true" },
    { onConflict: "org_id,key" }
  );
  ```
- Return JSON response with count of emails sent and skipped.
