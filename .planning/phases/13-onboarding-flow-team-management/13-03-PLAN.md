**NavLinks (`components/nav-links.tsx`):**

- Define admin-only routes. Per the user's locked decision:
  - Admin sees: Dashboard, Clients, Email Logs, Settings (including Team, Billing, Email Config)
  - Member sees: Dashboard, Clients, Email Logs — that's it
  - So Reminder Schedules, Email Templates, and Billing in the nav are admin-only.
- Filter the `NAV_ITEMS` array: hide items where `href` is in the admin-only list when `orgRole !== "admin"`.
- Admin-only nav items: `/schedules`, `/templates`, `/billing`.
- Member-visible nav items: `/dashboard`, `/clients`, `/email-logs`.
- The current NAV_ITEMS array has: `/dashboard`, `/clients`, `/schedules`, `/templates`, `/email-logs`, `/billing`.
- Add a `adminOnly?: boolean` flag to each NAV_ITEM, or define a set of admin-only hrefs and filter at render time. The filtering approach is cleaner:

```typescript
const ADMIN_ONLY_HREFS = new Set(["/schedules", "/templates", "/billing"]);

// In the component:
const visibleItems = NAV_ITEMS.filter(
  item => orgRole === "admin" || !ADMIN_ONLY_HREFS.has(item.href)
);
```

- Map over `visibleItems` instead of `NAV_ITEMS`.

**Settings page route protection (`app/(dashboard)/settings/page.tsx`):**
- Add at the top of the server component (before data fetching):

  const { orgRole } = await getOrgContext();

    redirect("/dashboard");

- Import `getOrgContext` from `@/lib/auth/org-context` and `redirect` from `next/navigation`.
- This silently redirects members who type `/settings` directly in the URL bar.

**Note:** The `/schedules` and `/templates` routes should also be protected for members.
