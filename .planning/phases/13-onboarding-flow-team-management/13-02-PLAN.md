**Team server actions (`app/actions/team.ts`):**

Create a new file with `"use server"` directive containing all team management server actions:

1. `getTeamMembers(): Promise<TeamMember[]>`:
   - Call `getOrgContext()` to get orgId. Only admins should call this (UI enforces, but no server-side role check needed since viewing team is a read operation).
   - Use admin client to query `user_organisations` WHERE `org_id = orgId`, ordered by `created_at ASC`.
   - For each membership, call `admin.auth.admin.getUserById(userId)` to resolve email (same pattern as `sendPaymentFailedEmail` in `notifications.ts` — PostgREST FK joins to auth.users don't work).
   - Also query `invitations` WHERE `org_id = orgId AND accepted_at IS NULL AND expires_at > now()` to get pending invites.
   - Return combined array of `TeamMember` objects: `{ id, userId?, email, role, status: "active" | "pending", joinedAt?, expiresAt? }`.

2. `sendInvite(email: string, role: "admin" | "member"): Promise<{ error?: string }>`:
   - Call `getOrgContext()`, verify `orgRole === "admin"`. Return error if not admin.
   - **Seat limit check:** Query `user_organisations` count for this org. Query `org.user_count_limit` from `organisations`. If count >= limit (and limit is not null), return `{ error: "Team is at capacity. Upgrade your plan to add more members." }`.
   - Also count pending invitations (not expired, not accepted) and add to the seat count.
   - Check if email already has a `user_organisations` row for this org (already a member). Return error if so.
   - Check if there's already a pending invite for this email+org. Return error if so: "An invite has already been sent to this email."
   - Generate token: `crypto.randomBytes(32).toString("hex")` (64-char hex string).
   - Hash token: `crypto.createHash("sha256").update(rawToken).digest("hex")`.
   - Set expiry: `new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)`.
   - INSERT into `invitations` via admin client: `{ org_id: orgId, email, role, token_hash: hash, expires_at: expiresAt.toISOString() }`.
   - Get org name from `organisations` table for the email content.
   - Get inviter email from `getOrgContext()` + `admin.auth.admin.getUserById()`.
   - Call `sendInviteEmail()` (defined in `notifications.ts`) with the raw token, email, org name, inviter email, role.
   - Return `{}` on success.

3. `removeTeamMember(targetUserId: string): Promise<{ error?: string }>`:
   - Call `getOrgContext()`, verify `orgRole === "admin"`. Return error if not admin.
   - Query target's role in `user_organisations`.
   - If target is an admin, count total admins for this org. If count <= 1, return `{ error: "Cannot remove the last admin. Promote another member to admin first." }` (TEAM-06).
   - DELETE from `user_organisations` WHERE `org_id = orgId AND user_id = targetUserId`.
   - Return `{}` on success.

4. `changeRole(targetUserId: string, newRole: "admin" | "member"): Promise<{ error?: string }>`:
   - Call `getOrgContext()`, verify `orgRole === "admin"`. Return error if not admin.
   - If demoting from admin to member, check admin count. If count <= 1, return `{ error: "Cannot demote the last admin." }` (TEAM-06 protection).
   - UPDATE `user_organisations` SET `role = newRole` WHERE `org_id = orgId AND user_id = targetUserId`.
   - Return `{}` on success.

5. `cancelInvite(inviteId: string): Promise<{ error?: string }>`:
   - Call `getOrgContext()`, verify `orgRole === "admin"`.
   - DELETE from `invitations` WHERE `id = inviteId AND org_id = orgId`.
   - Return `{}` on success.

6. `resendInvite(inviteId: string): Promise<{ error?: string }>`:
   - Call `getOrgContext()`, verify `orgRole === "admin"`.
   - Look up the invitation by id + org_id.
   - Generate a new token (new random bytes + new hash).
   - Update the invitation: new `token_hash`, new `expires_at` (7 days from now).
   - Resend the invite email with the new raw token.
   - Return `{}` on success.

**Accept invite actions (`app/(auth)/invite/accept/actions.ts`):**

Create with `"use server"` directive:

1. `validateInviteToken(token: string): Promise<{ valid: boolean; orgName?: string; inviterEmail?: string; role?: string; error?: string }>`:
   - Hash the raw token: `crypto.createHash("sha256").update(token).digest("hex")`.
   - Use admin client (service-role) to query `invitations` WHERE `token_hash = hash AND accepted_at IS NULL AND expires_at > now()`. Using admin client because the accepting user has no org_id in JWT yet (they're not a member), so RLS on `invitations` would block the query.
   - If no row found, return `{ valid: false, error: "This invite link is invalid or has expired." }`.
   - Query `organisations` for the org name.
   - Return `{ valid: true, orgName, role: invite.role }`.

2. `acceptInvite(token: string): Promise<{ orgSlug?: string; error?: string }>`:
   - Hash the token and look up the invitation (same as validateInviteToken).
   - Get the current authenticated user: `createClient()` → `supabase.auth.getUser()`. If not authenticated, return `{ error: "You must be signed in to accept this invite." }`.
   - Check if user already belongs to any org: query `user_organisations` WHERE `user_id = user.id`. If exists, return `{ error: "You already belong to another organisation. You must leave your current organisation before joining a new one." }`.
   - INSERT into `user_organisations`: `{ user_id: user.id, org_id: invite.org_id, role: invite.role }` via admin client.
   - UPDATE `invitations` SET `accepted_at = now()` WHERE `id = invite.id` via admin client.
   - Query `organisations.slug` for the accepted org.
   - Return `{ orgSlug: org.slug }`.

**Accept invite page (`app/(auth)/invite/accept/page.tsx`):**

This is a client component ("use client") since it needs interactive state:

**Auth state handling:**
- On mount, check `supabase.auth.getUser()` via browser client.
- If authenticated: show the accept page with "Accept & Join" button directly.
- If NOT authenticated: show "Sign in to accept this invite" with an email input and magic link button. The magic link `emailRedirectTo` should include the token: `${NEXT_PUBLIC_APP_URL}/invite/accept?token=${token}` so the user returns to the accept page after auth. Actually, route through `/auth/callback` for proper code exchange: `${NEXT_PUBLIC_APP_URL}/auth/callback`. After auth, the middleware will see the user has no org → redirect to `/onboarding`. That's wrong for invite flow.
- **Better approach for unauthenticated invitees:** Show a magic link form. After they click the magic link in their email, they arrive at `/auth/callback`. The callback exchanges the code, then redirects. Since the user has no org, middleware sends them to `/onboarding`. But they should go to `/invite/accept?token=...` instead.
- **Solution:** Store the invite token in a cookie or localStorage before sending the magic link. After auth, redirect to `/invite/accept?token=...`. Or: use `emailRedirectTo` that preserves the token.
- **Simplest correct solution:** Use `emailRedirectTo: "${NEXT_PUBLIC_APP_URL}/auth/callback?next=/invite/accept?token=${encodeURIComponent(token)}"`. But this requires the auth callback to read a `next` param and redirect there.
- **Even simpler:** The auth callback in `app/(auth)/auth/callback/route.ts` needs a tiny addition. But that file is not in this plan's scope. Let me think of another way.
- **Pragmatic solution:** Don't use magic link for invite accept auth. Instead, show a message: "Please sign in first, then come back to this page." with a "Sign In" link to `/login`. The user signs in via the existing login page, then navigates back to the invite link (they can bookmark it or re-click it from email). This is slightly more friction but avoids modifying the auth callback.
- **Actually:** For invite accept, the user likely already has an account (they were invited by email — they may already use the platform). If they don't have an account, they need to create one. The existing login page handles both cases (magic link creates account if new). After login, they can re-click the invite link from their email.
- **Approach:** If unauthenticated, show the invite details (org name, role) and a prominent "Sign In to Accept" button that links to `/login`. After login, the user clicks the invite link from their email again and lands on the accept page as an authenticated user. This is the cleanest UX without modifying the auth callback.
- Add a note on the page: "After signing in, click the invite link in your email again to accept."
