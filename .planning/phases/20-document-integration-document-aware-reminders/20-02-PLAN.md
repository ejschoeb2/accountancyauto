  <action>
Create a new file `lib/documents/auto-records-received.ts` exporting one async function:

```typescript
export async function checkAndAutoSetRecordsReceived(
  supabase: SupabaseClient,
  clientId: string,
  filingTypeId: string,
  orgId: string
): Promise<boolean>
```

Implementation:
1. Fetch `filing_document_requirements` for this `filingTypeId`, joining `document_types(label)`. Filter `is_mandatory = true`.
2. Fetch `client_document_checklist_customisations` for this `clientId` + `filingTypeId`. Build a `Map<documentTypeId, is_enabled>`.
3. Apply customisations: the effective mandatory list = requirements filtered by `customisationMap.get(doc_type_id) !== false` (default enabled when no row).
4. If effective mandatory list is empty, return `false` (nothing to satisfy).
5. Fetch `client_documents` for `clientId` + `filingTypeId` with `classification_confidence` IN `('high', 'medium')`.
6. Build a `Set<string>` of satisfied `document_type_id` values.
7. Check `allSatisfied = mandatoryItems.every(item => satisfiedSet.has(item.document_type_id))`.
8. If not all satisfied, return `false`.
9. Fetch the current `clients.records_received_for` array for `clientId` (select single row).
10. If `filingTypeId` is already in the array, return `false` (idempotent — already set).
11. Append `filingTypeId` to the array and UPDATE `clients SET records_received_for = updatedArray WHERE id = clientId`.
12. Return `true`.

IMPORTANT — pitfall from RESEARCH.md: This function uses a read-then-write pattern. The worst case is an idempotent write (same value already present), which is safe. Do NOT use a revoke-then-insert pattern here.

IMPORTANT: Use the `supabase` parameter passed in — do NOT create a new client inside this function. The caller decides whether to use an authenticated or service client.
