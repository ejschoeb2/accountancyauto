  <name>Task 1: Create Stripe webhook route handler with idempotency</name>
  <files>

  <action>
    1. Create `app/api/stripe/webhook/route.ts`:
       - Export `POST` handler (not GET -- webhooks are POST)
       - CRITICAL: Use `await req.text()` NOT `req.json()` to get raw body BEFORE signature verification
       - Get `stripe-signature` header from request
       - Verify signature using `stripe.webhooks.constructEvent(body, signature, process.env.STRIPE_WEBHOOK_SECRET!)`
       - Return 400 if signature invalid
       - IDEMPOTENCY: Use admin client (service_role) to check `processed_webhook_events` for `event.id`
         - If found, return 200 immediately (already processed)
         - If not found, insert into `processed_webhook_events` with `event_id`, `event_type`, `processed_at`
       - Dispatch to event-specific handlers via switch on `event.type`:
         - `checkout.session.completed` -> handleCheckoutSessionCompleted
         - `customer.subscription.updated` -> handleSubscriptionUpdated
         - `customer.subscription.deleted` -> handleSubscriptionDeleted
         - `invoice.payment_failed` -> handleInvoicePaymentFailed
         - default: log unhandled event type
       - Wrap handler dispatch in try/catch. On error, log but still return 200 (event already marked processed, don't want Stripe to retry and cause double-processing)
       - Return `NextResponse.json({ received: true })` at end

    2. Create `lib/stripe/webhook-handlers.ts` with 4 exported handler functions:

    a) `handleCheckoutSessionCompleted(session: Stripe.Checkout.Session, supabase: SupabaseClient)`:
       - Extract `org_id` and `plan_tier` from `session.metadata`
       - If missing metadata, log error and return (non-fatal)
       - Extract `customerId = session.customer as string` and `subscriptionId = session.subscription as string`
       - Import stripe client, call `stripe.subscriptions.retrieve(subscriptionId)` to get current status
       - Import `getPlanByTier` from plans.ts to get limits for the tier
       - Update organisations row where id = org_id:
         - stripe_customer_id = customerId
         - stripe_subscription_id = subscriptionId
         - stripe_price_id = subscription.items.data[0]?.price.id (if available)
         - subscription_status = subscription.status (cast to match enum: 'active', 'trialing', etc.)
         - plan_tier = planTier (from metadata)
         - client_count_limit = limits from getPlanByTier
         - user_count_limit = limits from getPlanByTier
         - trial_ends_at = subscription.trial_end ? new Date(subscription.trial_end * 1000).toISOString() : null
         - updated_at = new Date().toISOString()
       - Log success

    b) `handleSubscriptionUpdated(subscription: Stripe.Subscription, supabase: SupabaseClient)`:
       - Find org by stripe_subscription_id = subscription.id (using admin client)
       - If no org found, log warning and return
       - Extract new price ID from subscription.items.data[0]?.price.id
       - Use `getPlanByPriceId` to determine new plan tier and limits
       - Update organisations row:
         - subscription_status = subscription.status
         - If plan changed (new price ID != current): update plan_tier, client_count_limit, user_count_limit, stripe_price_id
         - If subscription has cancel_at_period_end = true, keep status as-is (Stripe handles transition)
         - updated_at = new Date().toISOString()
       - Log update

    c) `handleSubscriptionDeleted(subscription: Stripe.Subscription, supabase: SupabaseClient)`:
       - Find org by stripe_subscription_id = subscription.id
       - If no org found, log warning and return
       - Update organisations row:
         - subscription_status = 'cancelled'
         - updated_at = new Date().toISOString()
       - Log cancellation

    d) `handleInvoicePaymentFailed(invoice: Stripe.Invoice, supabase: SupabaseClient)`:
       - Extract customer_id from invoice.customer (as string)
       - Find org by stripe_customer_id = customer_id
       - If no org found, log warning and return
       - Update organisations subscription_status to 'past_due' (Stripe sends this when payment fails)
       - Call `sendPaymentFailedEmail(orgId, supabase)` from lib/billing/notifications.ts (created in Task 2)
       - Log payment failure

    All handlers use the admin client (SupabaseClient with service_role) passed in from the webhook route. This bypasses RLS since webhooks have no user session.

    IMPORTANT: Do NOT handle `customer.subscription.created` -- use checkout.session.completed as sole provisioning trigger (per research: avoids race condition where subscription.created fires before checkout.session.completed).

  <action>
    Create `lib/billing/notifications.ts`:

    Export `async function sendPaymentFailedEmail(orgId: string, supabase: SupabaseClient)`:
    - Query organisations for org name where id = orgId
    - Query user_organisations for admin users (role = 'admin') joined with auth.users to get email addresses
      - NOTE: Since PostgREST FK join may not resolve auth.users, query user_organisations for admin user_ids, then use supabase.auth.admin.getUserById() for each admin (this is the admin client, so it has access)
    - For each admin user email:
      - Use Postmark directly (import ServerClient from 'postmark', use env var POSTMARK_SERVER_TOKEN) to send a simple email
      - Subject: "Payment failed for {org_name}"
      - Body (HTML): Simple, professional email:
        - "Hi,"
        - "We were unable to process your latest payment for {org_name}."
        - "Your account access may be restricted until payment is resolved."
        - "Please update your payment method:"
        - CTA button/link: "Update Payment Method" linking to `{NEXT_PUBLIC_APP_URL}/billing` (they can use "Manage billing" to get to Stripe Portal)
        - "If you have any questions, reply to this email."
        - Footer with app name
      - Body (Text): Plain text version of the same content
      - From: Use env var based sender (e.g., "Peninsula Accounting <noreply@phasetwo.uk>") -- keep it simple, use env vars directly since this is a system notification, not an org-specific reminder
    - Log the number of admin emails sent
    - Do not throw on individual email failures (log and continue to next admin)

    This function is called by handleInvoicePaymentFailed in the webhook handler. The webhook's idempotency check ensures this email is sent at most once per Stripe event, satisfying the requirement "re-delivering the same webhook event does not send a second email."

    NOTE: This is a system notification email, not a client reminder. It uses the platform's Postmark token (env var), not the org's token. The email goes to the org admin, not to clients.
