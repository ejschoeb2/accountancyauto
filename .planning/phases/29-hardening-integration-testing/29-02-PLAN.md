<objective>
Harden the Postmark inbound webhook handler against delivery retries by adding two-layer idempotency: a unique constraint on inbound_emails keyed by postmark_message_id, and a pre-insert duplicate check inside processAttachments.

Purpose: Postmark retries failed deliveries up to 10 times over ~10.5 hours. Without idempotency, each retry creates duplicate inbound_emails rows and, because processAttachments fires on each, duplicate client_documents rows. The 200 response is already returned before uploads complete (fire-and-forget) — the timeout risk is solved. This plan closes the duplicate-row gap.

Output: One migration (column + constraint) and targeted changes to the inbound handler.
</objective>

```sql
-- Phase 29 HRDN-02: Add postmark_message_id column and idempotency unique constraint
-- Prevents duplicate inbound_emails rows on Postmark delivery retries.
--
-- postmark_message_id is nullable (TEXT) for backwards compatibility — existing rows have no
-- MessageID (they were inserted before this column existed). The unique constraint uses
-- NULLS NOT DISTINCT so that NULL values do not satisfy the constraint (each NULL is unique).
-- This means only rows with a non-NULL postmark_message_id participate in deduplication.
--
-- The unique constraint scope is (org_id, postmark_message_id) — MessageID is globally unique
-- per Postmark, but scoping to org_id provides defence-in-depth and matches the table's
-- existing per-org data model.

ALTER TABLE inbound_emails
  ADD COLUMN IF NOT EXISTS postmark_message_id TEXT;

-- Add unique constraint (NULLS NOT DISTINCT requires Postgres 15+; Supabase uses PG15+)
ALTER TABLE inbound_emails
  DROP CONSTRAINT IF EXISTS uq_inbound_emails_org_message_id;

ALTER TABLE inbound_emails
  ADD CONSTRAINT uq_inbound_emails_org_message_id
  UNIQUE NULLS NOT DISTINCT (org_id, postmark_message_id);
```

```typescript
const { data: inboundEmail, error: insertError } = await supabase
  .from('inbound_emails')
  .insert({
    org_id: orgId,
    client_id: client?.id || null,
```

```typescript
    console.warn('[Postmark Inbound] Duplicate delivery detected (idempotency), skipping:', messageId);
```

**Change 2 — processAttachments idempotency check + size guard:**

In `processAttachments`, inside the `for (const attachment of attachments)` loop, after computing `sha256Hash` (around line 268), add:

```typescript
// Phase 29 HRDN-02: Size guard — skip oversized attachments rather than failing silently
const MAX_ATTACHMENT_BYTES = 25 * 1024 * 1024; // 25 MB (Postmark inbound limit)
if (fileBuffer.length > MAX_ATTACHMENT_BYTES) {
  console.warn('[Postmark Inbound] Attachment exceeds size limit, skipping:', {
    filename: attachment.Name,
    sizeBytes: fileBuffer.length,
    maxBytes: MAX_ATTACHMENT_BYTES,
  });
  continue;
}

// Phase 29 HRDN-02: Idempotency guard — skip if this exact attachment was already stored
// Keyed on (client_id, file_hash, source) — if a row exists, this is a Postmark retry
if (client?.id) {
  const { data: existingDoc } = await supabase
    .from('client_documents')
    .select('id')
    .eq('client_id', client.id)
    .eq('file_hash', sha256Hash)
    .eq('source', 'inbound_email')

    console.warn('[Postmark Inbound] Idempotency: duplicate attachment skipped', {
      filename: attachment.Name,
      existingDocumentId: existingDoc.id,
    });
    continue;
  }
}
```
