**Step A — Delete app/page.tsx content and replace with a redirect to the marketing route:**

Actually, the `(marketing)` route group resolves to the same URL as `app/page.tsx` — they both map to `/`. Having BOTH would cause a conflict. The solution: **delete the contents of `app/page.tsx` and replace it with a simple re-export or redirect**, OR simply delete `app/page.tsx` entirely and let `app/(marketing)/page.tsx` claim the `/` route.

The cleanest approach: **Delete `app/page.tsx`** (remove the file entirely). The `app/(marketing)/page.tsx` will now be the sole handler for `/`.

However, if deleting causes issues with Next.js routing (the (dashboard) route group may have its own page.tsx at a different path), simply emptying and re-pointing is fine. The safest approach: delete `app/page.tsx` and create `app/(marketing)/page.tsx` as the replacement.

**Step D — Update middleware to allow `/` in development:**

In `lib/supabase/middleware.ts`, modify Step 5 (no-slug handling) to allow the root path through without redirecting in development:

Current behaviour (Step 5, no slug, unauthenticated, dev):
```typescript
if (isDev) {
  const loginUrl = new URL("/login", request.url);
  const redirect = NextResponse.redirect(loginUrl, 307);
  copyCookies(supabaseResponse, redirect);
  return redirect;
}
```

Updated behaviour — add a check for exact root path BEFORE the no-slug redirect:
```typescript
// Allow exact root path "/" through for the marketing page (dev preview)
if (pathname === "/") {
  return supabaseResponse;
}
```

This check should be inserted at the TOP of Step 5 (before any no-slug logic runs), so that `GET /` always reaches the marketing page regardless of auth state or slug presence. In production, the marketing page is served from `phasetwo.uk` directly, so this dev shortcut does not affect production routing.

**Specifically:** Insert the root path check at the very beginning of the `if (!slug)` block (after line `if (!slug) {`), before the `if (user)` check:

```typescript
// Step 5: Handle no-slug cases
if (!slug) {
  // Allow the root marketing page through (dev preview + production root)
  if (pathname === "/") {
    return supabaseResponse;
  }

  if (user) {
    // ... existing authenticated no-slug logic
  }
```

This means:
- `localhost:3000/` → marketing page (dev)
- `localhost:3000/dashboard?org=acme` → normal app flow (has slug from ?org=)
- `app.phasetwo.uk/` → production middleware never reaches this code (slug is extracted from subdomain or host)
- `phasetwo.uk/` → marketing page (production root deployment)

Also, for authenticated users on the root in development: they will see the marketing page when visiting `/` directly. This is acceptable — the dashboard is at `/dashboard?org=slug`. If you want authenticated dev users at `/` to redirect to their org dashboard, you could limit the bypass to unauthenticated only — but per the research recommendation, allowing it universally is simpler and more useful for dev preview.
