<objective>
Create the JWT Custom Access Token Hook that injects org_id and org_role into every authenticated session's JWT, then rewrite all RLS policies to scope data access by org_id from the JWT claim.

Purpose: This is the security layer that makes multi-tenancy real. Without this, org_id is just a column. With this, every query is automatically filtered to the user's org and cross-tenant access is impossible.

Output: Two migration files — one for the JWT hook function, one for the RLS policy rewrite.
</objective>

  <name>Task 1: Create Custom Access Token Hook</name>
  <files>supabase/migrations/20260219000004_create_jwt_hook.sql</files>
  <action>
Create a Supabase Custom Access Token Hook function. This is a PL/pgSQL function that Supabase Auth calls every time it issues a JWT. The function looks up the user's org membership and injects `org_id` and `org_role` into `app_metadata`.

**CRITICAL DECISION (locked):** Use `app_metadata`, NOT `user_metadata`. `user_metadata` is client-writable and would allow users to change their own org_id. `app_metadata` requires service role to modify.

**Create the hook function:**

```sql
-- Custom Access Token Hook
-- Called by Supabase Auth on every token refresh/issue
-- Injects org_id and org_role from user_organisations into app_metadata
CREATE OR REPLACE FUNCTION public.custom_access_token_hook(event jsonb)
RETURNS jsonb
LANGUAGE plpgsql
STABLE
AS $$
DECLARE
  claims jsonb;
  user_org record;
BEGIN
  -- Extract the claims from the event
  claims := event->'claims';

  -- Look up the user's org membership
  -- If user belongs to multiple orgs, pick the first one (for now — Phase 13 will add org switching)
  SELECT uo.org_id, uo.role
  INTO user_org
  FROM user_organisations uo
  WHERE uo.user_id = (event->>'user_id')::uuid
  ORDER BY uo.created_at ASC
  LIMIT 1;

  -- If user has an org membership, inject into app_metadata
  IF user_org IS NOT NULL THEN
    claims := jsonb_set(
      claims,
      '{app_metadata}',
      COALESCE(claims->'app_metadata', '{}'::jsonb) ||
      jsonb_build_object(
        'org_id', user_org.org_id::text,
        'org_role', user_org.role::text
      )
    );
  END IF;

  -- Return the modified event with updated claims
  event := jsonb_set(event, '{claims}', claims);
  RETURN event;
END;
$$;
```

**Grant permissions for the hook to work:**
```sql
-- Supabase Auth calls hooks as the supabase_auth_admin role
GRANT USAGE ON SCHEMA public TO supabase_auth_admin;
GRANT EXECUTE ON FUNCTION public.custom_access_token_hook TO supabase_auth_admin;

-- The hook needs to read user_organisations
GRANT SELECT ON TABLE public.user_organisations TO supabase_auth_admin;

-- Revoke execute from other roles (security hardening)
REVOKE EXECUTE ON FUNCTION public.custom_access_token_hook FROM authenticated, anon, public;
```

**IMPORTANT NOTE in comments:** After applying this migration, the user must enable the hook in the Supabase Dashboard:
- Go to Authentication > Hooks (under Configuration)
- Enable "Custom Access Token Hook"
- Select the schema `public` and function `custom_access_token_hook`
- Save

This step cannot be automated via SQL — it requires dashboard configuration.

Also create a helper function for RLS policies to extract org_id from JWT (avoids repeating the JSONPath in every policy):
```sql
CREATE OR REPLACE FUNCTION public.auth_org_id()
RETURNS uuid
LANGUAGE sql
STABLE
AS $$
  SELECT COALESCE(
    (current_setting('request.jwt.claims', true)::jsonb -> 'app_metadata' ->> 'org_id')::uuid,
    '00000000-0000-0000-0000-000000000000'::uuid
  );
$$;
```

Grant execute to authenticated role:
```sql
GRANT EXECUTE ON FUNCTION public.auth_org_id TO authenticated;
```

The COALESCE to a zero UUID ensures that if the claim is missing, the function returns a UUID that matches nothing (rather than NULL, which would cause unexpected behavior in equality checks).

<task type="auto">
  <name>Task 2: Rewrite all RLS policies to use org_id scoping</name>
  <files>supabase/migrations/20260219000005_rewrite_rls_policies.sql</files>
  <action>
Create a migration that drops ALL existing `USING(true)` authenticated policies and replaces them with org_id-scoped policies. Service role policies stay as-is (they need full access for cron jobs).

**DECISION (locked):** Phased activation by table group — enable on core tables first, then secondary tables. However, since this is a single migration file, we activate all at once but organize the SQL in logical groups for readability.

**DECISION (locked):** Policies use JWT claims via `auth_org_id()` helper function (calls `auth.jwt() -> 'app_metadata' ->> 'org_id'`).

**Policy pattern for data tables (all 15 tables with org_id):**

For each table, drop all existing authenticated policies, then create:
```sql
-- SELECT: user can only read their org's data
CREATE POLICY "{table}_select_org" ON {table}
  FOR SELECT TO authenticated
  USING (org_id = auth_org_id());

-- INSERT: user can only insert into their org
CREATE POLICY "{table}_insert_org" ON {table}
  FOR INSERT TO authenticated
  WITH CHECK (org_id = auth_org_id());

-- UPDATE: user can only update their org's data
CREATE POLICY "{table}_update_org" ON {table}
  FOR UPDATE TO authenticated
  USING (org_id = auth_org_id())
  WITH CHECK (org_id = auth_org_id());

-- DELETE: user can only delete their org's data
CREATE POLICY "{table}_delete_org" ON {table}
  FOR DELETE TO authenticated
  USING (org_id = auth_org_id());
```

**Tables to apply org-scoped policies (Group 3 — System):**
14. `app_settings`
15. `locks`

**Tables with special policies (no org_id):**
- `filing_types` — keep `USING(true)` for SELECT (global read), no INSERT/UPDATE/DELETE for authenticated (reference data)
- `bank_holidays_cache` — keep `USING(true)` for SELECT (global read), no INSERT/UPDATE/DELETE for authenticated

**Tables with special policies (multi-tenant):**
- `organisations` — authenticated users can SELECT only their own org: `USING (id = auth_org_id())`
- `user_organisations` — authenticated users can SELECT only their org's members: `USING (org_id = auth_org_id())`
- `invitations` — authenticated users can manage only their org's invitations: `USING (org_id = auth_org_id())`

**DROP existing policies first.** For each table, drop all existing authenticated policies by name. Reference the policy names from the original migrations:
- `"Authenticated users can read {table}"` (SELECT)
- `"Authenticated users can modify {table}"` (ALL)
- `"Authenticated users can insert {table}"` (INSERT)
- `"Authenticated users can update {table}"` (UPDATE)
- `"Authenticated users can manage {table}"` (ALL variants)

Use `DROP POLICY IF EXISTS` for safety.

**DO NOT touch service_role policies** — cron jobs and admin operations use service_role which bypasses RLS entirely. Those policies stay as `USING(true)`.

**Special notes:**
- The `clients` table has separate SELECT/INSERT/UPDATE policies (no ALL). Keep the same pattern.
- For `locks`, the policy ensures each org can only see/modify its own locks. Lock keys will be made org-scoped in Plan 03.
- For `filing_types` and `bank_holidays_cache`, drop any write policies for authenticated and keep only SELECT USING(true).
