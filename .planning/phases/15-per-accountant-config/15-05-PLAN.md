Implement new-user seeding on invite acceptance. When a member joins an org, clone the admin's templates and schedules to give them a working reminder setup immediately.

Purpose: Without seeding, new members start with zero templates and zero schedules, making reminders non-functional until they manually create everything. Cloning from the admin gives them a fully working setup from day one.

Output: New seeding utility module and updated invite acceptance flow.
</objective>

**Function signature:**
```typescript
export async function seedNewUserDefaults(
  userId: string,
  orgId: string
): Promise<void>
```

**Implementation steps:**

1. Create admin client (service-role, bypasses RLS — the new user has no org_id in JWT yet).

2. Find the org's earliest admin:
```typescript
const { data: adminMembership } = await admin
  .from('user_organisations')
  .select('user_id')
  .eq('org_id', orgId)
  .eq('role', 'admin')
  .order('created_at', { ascending: true })
  .limit(1)
  .single();

if (!adminMembership) {
  console.warn(`[seed-new-user] No admin found for org ${orgId}, skipping seeding`);
  return;
}
const adminId = adminMembership.user_id;
```

3. **Clone email_templates:**
```typescript
const { data: adminTemplates } = await admin
  .from('email_templates')
  .select('*')
  .eq('org_id', orgId)
  .eq('owner_id', adminId);

// Build old_id -> new_id mapping for schedule_steps FK remapping
const templateIdMap = new Map<string, string>();

if (adminTemplates && adminTemplates.length > 0) {
  for (const template of adminTemplates) {
    const { id: oldId, created_at, updated_at, ...rest } = template;
    const { data: inserted, error } = await admin
      .from('email_templates')
      .insert({ ...rest, owner_id: userId })
      .select('id')
      .single();

    if (inserted) {
      templateIdMap.set(oldId, inserted.id);
    } else if (error) {
      console.warn(`[seed-new-user] Failed to clone template ${oldId}:`, error.message);
    }
  }
}
```

Note: Insert templates one by one (not bulk) to capture the returned IDs for FK remapping. The template count per admin is typically 3-10, so this is fine.

4. **Clone schedules:**
```typescript
const { data: adminSchedules } = await admin
  .from('schedules')
  .select('*')
  .eq('org_id', orgId)
  .eq('owner_id', adminId);

const scheduleIdMap = new Map<string, string>();

if (adminSchedules && adminSchedules.length > 0) {
  for (const schedule of adminSchedules) {
    const { id: oldId, created_at, updated_at, ...rest } = schedule;
    const { data: inserted, error } = await admin
      .from('schedules')
      .insert({ ...rest, owner_id: userId })
      .select('id')
      .single();

    if (inserted) {
      scheduleIdMap.set(oldId, inserted.id);
    } else if (error) {
      console.warn(`[seed-new-user] Failed to clone schedule ${oldId}:`, error.message);
    }
  }
}
```

5. **Clone schedule_steps with remapped FKs:**
```typescript
const { data: adminSteps } = await admin
  .from('schedule_steps')
  .select('*')
  .eq('org_id', orgId)
  .eq('owner_id', adminId);

if (adminSteps && adminSteps.length > 0) {
  const newSteps = adminSteps
    .map(({ id, created_at, updated_at, ...step }) => {
      const newScheduleId = scheduleIdMap.get(step.schedule_id);
      const newTemplateId = templateIdMap.get(step.email_template_id);

      // Skip if either FK couldn't be remapped
      if (!newScheduleId || !newTemplateId) {
        console.warn(`[seed-new-user] Skipping step ${id}: missing FK remap (schedule: ${!!newScheduleId}, template: ${!!newTemplateId})`);
        return null;
      }

      return {
        ...step,
        owner_id: userId,
        schedule_id: newScheduleId,
        email_template_id: newTemplateId,
      };
    })
    .filter(Boolean);

  if (newSteps.length > 0) {
    const { error } = await admin
      .from('schedule_steps')
      .insert(newSteps);

    if (error) {
      console.warn(`[seed-new-user] Failed to clone schedule steps:`, error.message);
    }
  }
}
```

6. **Do NOT clone schedule_client_exclusions** — the new user starts with no clients, so there's nothing to exclude. Exclusions are set up per-client after the user has clients assigned.

7. **Do NOT clone app_settings** — the user starts with org defaults (the fallback pattern in settings reads handles this). If the user wants different settings, they change them on their settings page.

**Error handling:** Wrap the entire function in try/catch. Log errors with `console.error` but do NOT throw — seeding failure is non-fatal. The user is already in the org.
