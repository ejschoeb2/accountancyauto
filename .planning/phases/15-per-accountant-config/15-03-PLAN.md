**app/api/cron/reminders/route.ts changes:**

Inside the `for (const org of orgs)` loop, after the Postmark token check:

1. Fetch org members: query `user_organisations` for all users in this org
2. Loop over each member and call a new `processRemindersForUser(adminClient, org, member.user_id)` instead of `processReminders(adminClient, org)`
3. Accumulate results from each user call

Update the result interface to optionally include user-level breakdown:
```typescript
interface OrgResult {
  org: string;
  org_id: string;
  queued: number;
  rolled_over: number;
  errors: string[];
  skipped_wrong_hour: boolean;
  error?: string;
  users_processed?: number;
}
```

```typescript
// Inside the org loop:
const { data: members } = await adminClient
  .from('user_organisations')
  .select('user_id')
  .eq('org_id', org.id);

let orgQueued = 0;
let orgRolledOver = 0;
const orgErrors: string[] = [];
let orgSkippedWrongHour = true; // Start true, set false if any user processes

for (const member of members ?? []) {
  try {
    const userResult = await processRemindersForUser(adminClient, org, member.user_id);
    orgQueued += userResult.queued;
    orgRolledOver += userResult.rolled_over;
    orgErrors.push(...userResult.errors);
    if (!userResult.skipped_wrong_hour) orgSkippedWrongHour = false;
  } catch (error) {
    const message = error instanceof Error ? error.message : 'Unknown error';
    orgErrors.push(`[user:${member.user_id}] ${message}`);
  }
}
```

**lib/reminders/scheduler.ts changes:**

1. Rename `processReminders` to `processRemindersForUser`. Update the signature to accept an additional `userId: string` parameter.

2. Change the lock key from `cron_reminders_${org.id}` to `cron_reminders_${org.id}_${userId}` — per-user lock prevents parallel processing of the same user.

3. **Send hour check**: Replace the org-level send hour read with a per-user fallback:
```typescript
// Try user-specific send hour first
const { data: userSendHourRow } = await supabase
  .from('app_settings')
  .select('value')
  .eq('org_id', org.id)
  .eq('user_id', userId)
  .eq('key', 'reminder_send_hour')
  .maybeSingle();

let globalSendHour: number;
if (userSendHourRow) {
  globalSendHour = parseInt(userSendHourRow.value, 10);
} else {
  // Fallback to org-level default
  const { data: orgSendHourRow } = await supabase
    .from('app_settings')
    .select('value')
    .eq('org_id', org.id)
    .is('user_id', null)
    .eq('key', 'reminder_send_hour')
    .maybeSingle();
  globalSendHour = orgSendHourRow ? parseInt(orgSendHourRow.value, 10) : 9;
}
```

4. **Custom schedule hour check**: Add `.eq('owner_id', userId)` to the custom schedule hour query so it only checks this user's custom schedules.

5. **Queue building**: Pass `userId` to both `buildReminderQueue` and `buildCustomScheduleQueue` (see Task 2).

6. **Due reminders fetch**: Filter `reminder_queue` by joining through `clients!inner(*)` and adding a filter on `clients.owner_id = userId`. This ensures each user's cron iteration only processes their own clients' reminders. Use the PostgREST nested filter pattern:
```typescript
const { data: allDueReminders } = await supabase
  .from('reminder_queue')
  .select('*, clients!inner(*), filing_types(*)')
  .eq('org_id', org.id)
  .eq('clients.owner_id', userId)
  .eq('send_date', today)
  .eq('status', 'scheduled');
```

7. **Schedule/template fetch in template rendering loop**: Add `.eq('owner_id', userId)` to the schedule, schedule_steps, and email_template queries that resolve templates for each reminder. This ensures we use the user's own templates, not another user's.

8. **Rollover logic**: Also scope past deadline queries by `clients.owner_id = userId`.

9. **Template rendering context**: Replace the hardcoded `accountant_name: 'PhaseTwo'` with a value from the user's email settings. Query the user's `email_sender_name` from app_settings (with org fallback) and use it. If not available, fall back to 'PhaseTwo'.

Export `processRemindersForUser` (not `processReminders`). The old name can be kept as a wrapper if needed for backwards compatibility, but the cron entry point should call the new function directly.

**buildReminderQueue changes:**

1. Add optional `ownerId?: string` parameter to the function signature. When provided, all resource queries are scoped by owner_id.

2. **Filing assignments query**: Add `.eq('clients.owner_id', ownerId)` to the `client_filing_assignments` query (which joins clients). This ensures only this user's clients are processed.

3. **Schedules query**: Add `.eq('owner_id', ownerId)` to the schedules fetch. Each user has their own filing schedules.

4. **Schedule steps query**: Add `.eq('owner_id', ownerId)` to the schedule_steps fetch. Steps belong to the user's schedules.

5. **Email templates query**: Add `.eq('owner_id', ownerId)` to the email_templates fetch.

6. **Schedule client exclusions query**: Add `.eq('owner_id', ownerId)` to the exclusions fetch. Note: schedule_client_exclusions now has owner_id (from Plan 01 migration). The exclusion should match the user's own exclusions on their own schedules.

All these filters are only added when `ownerId` is provided. This preserves backwards compatibility for `rebuildQueueForClient` which calls this from server actions (where RLS already handles scoping via the authenticated user's session).

Actually, `rebuildQueueForClient` uses the service-role client when called from cron, but the authenticated client when called from server actions. For server actions, RLS handles the scoping. For cron, the `ownerId` parameter should be passed. However, `rebuildQueueForClient` currently does NOT have per-user scoping — it rebuilds for all schedules/templates in the org. After this change, it should also accept an optional `ownerId` to scope resources. BUT: `rebuildQueueForClient` is called from server actions where the user is already authenticated and RLS filters their resources. So the current behavior is already correct for server actions. For now, leave `rebuildQueueForClient` unchanged — it works correctly because RLS on schedules/templates scopes to the user's own resources in server action context.

This is the correct interpretation per the research document: "After phase 15, a user's custom schedule should only fire for clients they own."
