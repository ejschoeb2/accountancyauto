Create two migration files.

**Migration 1: `20260222100001_add_owner_id_to_resource_tables.sql`**

Add `owner_id UUID REFERENCES auth.users(id)` to four tables: `email_templates`, `schedules`, `schedule_steps`, `schedule_client_exclusions`. Follow the exact pattern from `20260222000001_add_owner_id_and_drop_user_count_limit.sql`:

1. ALTER TABLE ... ADD COLUMN IF NOT EXISTS owner_id UUID REFERENCES auth.users(id)
2. Backfill with COALESCE (earliest admin in org > earliest member > any auth user) WHERE owner_id IS NULL
3. ALTER TABLE ... ALTER COLUMN owner_id SET NOT NULL
4. CREATE INDEX IF NOT EXISTS idx_{table}_owner_id ON {table}(owner_id)

Repeat for all four tables. Add COMMENT ON COLUMN for each.

**Migration 2: `20260222100002_add_user_id_to_app_settings.sql`**

1. Add nullable `user_id UUID REFERENCES auth.users(id)` to app_settings
2. Drop the old unique constraint: `ALTER TABLE app_settings DROP CONSTRAINT IF EXISTS app_settings_org_id_key_unique`
3. Add new unique constraint with NULLS NOT DISTINCT: `ALTER TABLE app_settings ADD CONSTRAINT app_settings_org_user_key_unique UNIQUE NULLS NOT DISTINCT (org_id, user_id, key)`
4. Create index: `CREATE INDEX IF NOT EXISTS idx_app_settings_user_id ON app_settings(user_id)`
5. Do NOT backfill user_id — existing rows remain NULL (they are org-level defaults, which is correct)
6. Do NOT set NOT NULL — user_id must remain nullable (NULL = org-level default, UUID = user override)

CRITICAL: The `NULLS NOT DISTINCT` modifier (PostgreSQL 15+) ensures that `(org_id, NULL, key)` is truly unique. Without it, duplicate org-level default rows would be allowed.
  </action>

  <action>
Create migration `20260222100003_rewrite_resource_tables_rls.sql` that rewrites RLS policies on five tables: `email_templates`, `schedules`, `schedule_steps`, `schedule_client_exclusions`, `app_settings`.

Follow the exact pattern from `20260222000002_rewrite_clients_rls_with_owner_scoping.sql`.

**For email_templates, schedules, schedule_steps, schedule_client_exclusions:**

Drop old org-scoped policies (created by `20260219000005_rewrite_rls_policies.sql`):
```sql
DROP POLICY IF EXISTS "{table}_select_org" ON {table};
DROP POLICY IF EXISTS "{table}_insert_org" ON {table};
DROP POLICY IF EXISTS "{table}_update_org" ON {table};
DROP POLICY IF EXISTS "{table}_delete_org" ON {table};
```

Create new owner-scoped policies per table:
- `{table}_select_admin`: FOR SELECT TO authenticated USING (org_id = auth_org_id() AND auth_org_role() = 'admin')
- `{table}_select_member`: FOR SELECT TO authenticated USING (org_id = auth_org_id() AND owner_id = auth.uid())
- `{table}_insert_org`: FOR INSERT TO authenticated WITH CHECK (org_id = auth_org_id() AND owner_id = auth.uid())
- `{table}_update_admin`: FOR UPDATE TO authenticated USING/WITH CHECK (org_id = auth_org_id() AND auth_org_role() = 'admin')
- `{table}_update_member`: FOR UPDATE TO authenticated USING/WITH CHECK (org_id = auth_org_id() AND owner_id = auth.uid())
- `{table}_delete_admin`: FOR DELETE TO authenticated USING (org_id = auth_org_id() AND auth_org_role() = 'admin')
- `{table}_delete_member`: FOR DELETE TO authenticated USING (org_id = auth_org_id() AND owner_id = auth.uid())

**For app_settings:**

Drop old org-scoped policies:
```sql
DROP POLICY IF EXISTS "app_settings_select_org" ON app_settings;
DROP POLICY IF EXISTS "app_settings_insert_org" ON app_settings;
DROP POLICY IF EXISTS "app_settings_update_org" ON app_settings;
DROP POLICY IF EXISTS "app_settings_delete_org" ON app_settings;
```

app_settings RLS stays org-scoped (NOT owner-scoped) — the user_id filtering is done in application code, not RLS. This is because:
- Org-level defaults (user_id IS NULL) need to be readable by all users in the org
- User-specific rows need to be readable by the owning user AND admins
- The fallback pattern (read user row, then org default) requires seeing both

Create new policies:
- `app_settings_select_org`: FOR SELECT TO authenticated USING (org_id = auth_org_id())
- `app_settings_insert_org`: FOR INSERT TO authenticated WITH CHECK (org_id = auth_org_id())
- `app_settings_update_org`: FOR UPDATE TO authenticated USING (org_id = auth_org_id()) WITH CHECK (org_id = auth_org_id())
- `app_settings_delete_org`: FOR DELETE TO authenticated USING (org_id = auth_org_id())

Note: app_settings policies stay the same policy names (drop and recreate) because the logic is unchanged — just confirming they exist after migration. If they already have the right shape, this is defensive (drop+recreate is safe).

Leave service_role policies untouched (they exist from earlier migrations and grant full access for cron).

Include a validation block (same pattern as Quick Task 5) to verify all expected policies exist and old ones are gone.
